## 一、项目概述

### 1.1 项目目标

创建一个景区导览数字人 C 端前端系统，包含两种模式：

- **主对话模式**（Markdown 文档定义）：数字人对话、音频播放、字幕显示、知识画轴、历史记录、满意度评分
- **路线导览模式**（Word 文档定义）：左侧数字人讲解 + 右侧路线图 + 景点卡片

### 1.2 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| 前端框架 | React 18 | UI 渲染 |
| 状态管理 | Zustand | 全局状态 |
| 2D 渲染 | pixi.js | Live2D 渲染引擎 |
| Live2D | pixi-live2d-display | Live2D Cubism 插件 |
| 后端 | Node.js + Express | 回环服务 |
| 构建工具 | Vite | 开发和构建 |

### 1.3 两个文档关系

| 文档 | 内容 | 模式 |
|------|------|------|
| 景区导览数字人C端前端开发方案v1.0.md | 完整数字人系统：状态机、WebSocket、Live2D、音频、字幕、控制区、知识画轴、地图画轴、历史记录、满意度 | 主对话模式 |
| Route Guide Frontend Specification.docx | 路线导览专属页面：左侧数字人+讲解卡，右侧路线图+景点卡 | 路线导览模式 |

**关键规则**：当两个文件内容不同时，以 Word 文档为准。


**Actions**：
- `transitionTo(newState)` - 状态切换
- `startThinking()` / `stopThinking()` - Thinking 控制
- `openPanel(responseId, segmentId, type, payload)` - 打开画轴
- `closePanel(isManual)` - 关闭画轴
- `clearResponse()` - 清理响应

#### 3.2 `messageStore.js`（消息历史 + 字幕状态）

```javascript
{
  messages: [],              // 消息历史
  fullAnswer: '',            // 当前回答（token 累积）
  previousSentence: '',      // 上一句字幕
  currentSentence: '',       // 当前句字幕
  sentenceQueue: []          // 句子队列
}
```

**Actions**：
- `addToken(token)` - 添加 token
- `enqueueSentence(sentence)` - 添加句子到队列
- `dequeueSentence(segmentId)` - 取出句子更新字幕
- `finishResponse()` - 完成一轮回答

#### 3.3 `audioStore.js`（音频播放状态）

```javascript
{
  isPlaying: false,
  activeSegmentId: null,     // 正在接收音频的 segment
  playingSegmentId: null,    // 正在播放的 segment
  segmentQueues: {},         // {segmentId: [chunk1, chunk2, ...]}
  segmentStatus: {}          // {segmentId: 'receiving'|'ready'|'playing'|'done'}
}
```

**Actions**：
- `handleAudioStart(segmentId, sampleRate, channels)`
- `handleBinaryChunk(data)`
- `handleAudioEnd(segmentId)`
- `handlePlaybackStart(segmentId)`
- `handlePlaybackEnd(segmentId)`

#### 3.4 `routeStore.js`（路线导览模式状态）

```javascript
{
  isRouteMode: false,        // 是否处于路线导览模式
```

**Actions**：
- `enterRouteMode(payload)` - 进入路线导览模式
- `exitRouteMode()` - 退出路线导览模式
- `setActiveSpotId(spotId)` - 设置当前高亮景点

### 第四步：WebSocket 通信（D3-D4）

#### 4.1 `websocket.js` WebSocket 客户端类

**功能**：
- 连接管理（connect/disconnect）
- 心跳机制（35s 间隔）
- 自动重连（最多 5 次）
- 消息发送/接收
- 二进制 PCM16 处理

**超时配置**：
```javascript
const TIMEOUT = {
  wsConnect: 10000,        // WebSocket 连接超时 10s
  wsMessage: 60000,        // 消息间隔超时 60s
  wsHeartbeat: 35000,      // 心跳间隔 35s
  thinking: 15000,         // Thinking 超时 15s
  thinkingForce: 30000,    // Thinking 强制结束 30s
  audioSegmentGap: 10000,  // 音频段间隔 10s
  audioTotal: 120000       // 单轮最大播放 120s
};
```

#### 4.2 `useWebSocket.js` Hook

**消息处理**（按 Markdown 文档 6.1 节）：
- `stream_start` - 仅标记，不切换状态
- `token` - 累积到 fullAnswer
- `sentence` - 进入 sentenceQueue
- `audio_start` - 标记 activeSegmentId
- `audio_end` - 标记后端发送完毕
- `panel_open` - 打开画轴
- `panel_close` - 校验 responseId + segmentId 后关闭
- `satisfaction_request` - 显示满意度 UI

**发送问题**：
```javascript
const sendQuestion = (content) => {
  if (mainState === 'speaking' || mainState === 'thinking') return;
  const responseId = generateResponseId();
  setCurrentResponseId(responseId);
  startThinking();
  wsClient.send({ type: 'question', responseId, content });
  addMessage({ role: 'user', content, responseId });
};
```

### 第五步：Live2D 数字人（D4-D5）

#### 5.1 `Live2DViewer.js`

**技术栈**：pixi.js + pixi-live2d-display

**功能**：

- Idle/Tap/Flic 动作切换
- LipSync 口型同步

**LipSync 实现**：
```javascript
function updateLipSync(audioAnalyser, model) {
  const dataArray = new Uint8Array(audioAnalyser.frequencyBinCount);
  audioAnalyser.getByteFrequencyData(dataArray);
  const volume = dataArray.slice(0, 10).reduce((a, b) => a + b) / 10 / 255;
  model.internalModel.coreModel.setParameterValueById('ParamMouthOpenY', volume);
}
```

### 第八步：音频播放与字幕同步（D12-D14）

#### 8.1 `audioPlayer.js`

**功能**：
- PCM16 解码和播放

- segment 队列串行播放

#### 8.2 `useAudioPlayer.js` Hook

**播放流程**（MVP 阶段）：
1. 收到 `audio_start` → 标记 activeSegmentId
2. 收到 binary × N → 加入该 segment 的队列
3. 收到 `audio_end` → 标记该 segment 为 ready
4. 检查是否可以开始播放（串行，前一个 segment 播完）
5. 开始播放该 segment 的第一个 chunk → 主状态切换为 SPEAKING
6. 播放完所有 chunk → playback_end
7. 更新字幕 + 关闭画轴
8. 检查是否所有 segment 都播放完毕

### 第十二步：联调与超时兜底（D17-D18）

#### 12.1 超时规则

```javascript
const TIMEOUT = {
  wsConnect: 10000,        // WebSocket 连接超时 10s
  wsMessage: 60000,        // 消息间隔超时 60s
  wsHeartbeat: 35000,      // 心跳间隔 35s
  thinking: 15000,         // Thinking 超时 15s（显示提示）
  thinkingForce: 30000,    // Thinking 强制结束 30s
  audioSegmentGap: 10000,  // 音频段间隔 10s
  audioTotal: 120000       // 单轮最大播放 120s
};
```

#### 12.2 状态触发规则

**规则 1**：SPEAKING 由首段音频开始播放触发，不由 stream_start 触发

**规则 2**：audio_end 只表示后端发送完成，playback_end 才表示前端播放完成

**规则 3**：MVP 阶段音频段必须串行发送，不支持多个 segment 交错传输

**规则 4**：画轴自动关闭绑定 playback_end

**规则 5**：panel_close 必须校验 responseId + segmentId

**规则 6**：manuallyClosedSegmentIds 使用 responseId:segmentId 作为 key

**规则 7**：satisfaction_request 使用 lastCompletedResponseId

**规则 8**：MVP 阶段禁止在 Speaking / Thinking 中发起新问题

#### 12.3 联调检查清单

| 检查项 | 预期行为 | 异常处理 |
|--------|----------|----------|
| stream_start | 不触发 SPEAKING，仅标记 streamEndReceived=false | 仅标记 |
| 首个 audio chunk | 触发 SPEAKING + 关闭 Thinking | 状态机转换 |
| sentence 到达 | 进入队列，不更新字幕 | 等待音频 |
| audio_start | 标记 activeSegmentId | 初始化队列 |
| binary 到达 | 归属到 activeSegmentId | 无 segmentId 则丢弃 |
| audio_end | 标记后端发送完毕 | 等待 playback_end |
| playback_end | 更新字幕 + 关闭画轴 | 检查所有 segment |
| stream_end | 标记 streamEndReceived=true | 检查是否可 finishResponse |
| finishResponse | 必须同时满足 stream_end 和所有 playback_end | 不满足则等待 |
| segment 串行 | MVP 阶段不支持交错传输 | 后端必须串行 |
| panel_close | 校验 responseId+segmentId | 不匹配则忽略 |
| manuallyClosedKey | 使用 responseId:segmentId 格式 | 防止冲突 |
| satisfaction_request | 使用 lastCompletedResponseId | 避免被过滤 |
| Speaking 中提问 | MVP 拒绝 | 稳定版可打断 |
| Thinking 超时 | 15s 显示提示 | 30s 自动结束 |
| WebSocket 断开 | 自动重连 | 5 次失败报错 |

## 九、最终页面描述

最终页面应呈现为一个清新、明亮、易读的景区路线导览界面。

页面背景是柔和的浅玉绿渐变，不使用任何图片背景。左上角是一张宣纸质感的路线讲解卡片，清楚告诉游客"当前在哪里、下一站去哪里、预计多久到达"。左下方是七分身数字人，穿着新中式导览服饰，以温和手势指向右侧路线图。

右侧上方是主路线地图模块，展示本地水彩景区路线图、当前位置、下一站、预计用时和路线经过顺序。地图保持明亮清晰，不叠加深色遮罩。右侧下方是本路线主要景点摘要区，横向展示三个景点卡片，每张卡包含图片、名称和短描述。当前正在讲解的景点卡以朱砂橙边框和"当前讲解"标签高亮。

整体画面应像一张现代化的景区导览手册：清新、友好、文化感强，同时信息层级清楚，适合游客快速理解路线。
